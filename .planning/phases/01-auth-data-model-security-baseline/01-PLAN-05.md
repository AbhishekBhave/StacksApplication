# Phase 01 Plan 05: Auth/Env Gap Closure

## 1) Goal
Close the three Phase 01 auth/env gaps so Expo boots reliably with valid Supabase env values, and auth flows (`sign up/sign in` + `forgot password`) no longer fail with secret-key errors.

## 2) Scope
- In scope: env validation, Supabase client key-type guards, auth-flow error handling improvements, and verification commands for the three failed UAT items.
- Out of scope: new product features, schema/migration work, onboarding/route-guard behavior beyond what is needed to unblock auth testing.

## 3) Tasks
1. Harden Supabase client env bootstrap
   - Touchpoints: `expo/lib/supabase.ts`
   - Add explicit required checks for `EXPO_PUBLIC_SUPABASE_URL` and `EXPO_PUBLIC_SUPABASE_ANON_KEY` before `createClient`.
   - Fail fast with actionable error text (which key is missing and where to set it).

2. Block secret/service keys in client runtime
   - Touchpoints: `expo/lib/supabase.ts`
   - Add key-format guard to reject `sb_secret_*` and service-role-like values in `EXPO_PUBLIC_SUPABASE_ANON_KEY`.
   - Return a clear configuration error explaining that only anon/publishable keys are valid in Expo client code.

3. Add startup env precheck for local dev
   - Touchpoints: `expo/package.json` (script), `expo/scripts/check-env.js` (new)
   - Implement `check-env.js` that validates required env vars and key type.
   - Wire as `prestart` (and optionally `pretest`) so missing/misconfigured env fails before app boot.

4. Improve auth flow error mapping for key misconfiguration
   - Touchpoints: `expo/components/AuthForm.tsx`, `expo/app/(auth)/forgot-password.tsx`
   - Map forbidden/invalid-key backend responses to friendly, actionable copy (for example: "Client is using a secret key; set anon/publishable key and restart Expo").
   - Keep existing validation behavior for other auth errors.

5. Update local env guidance and operator actions
   - Touchpoints: `expo/.env.example`, repo setup docs
   - Ensure `.env.example` uses anon key placeholder (not secret).
   - Document: run Expo from `expo/`, restart with cache clear after env edits, rotate/revoke previously exposed secret key in Supabase dashboard.

## 4) Verification
Run from repo root unless noted.

- `cd expo && node scripts/check-env.js`
  - Expected: exits 0 when env is valid; exits non-zero with clear message when URL/key missing or key looks secret/service-role.
- `cd expo && npm run start -- --clear`
  - Expected: Expo boots without `supabaseUrl is required` or missing-env red-screen errors.
- `cd expo && npm test -- --ci`
  - Expected: existing smoke test remains green (no regression from env/script changes).
- Manual UAT:
  - Sign up and sign in from `/(auth)`:
    - Expected: no "Forbidden use of secret API key"; session established.
  - Forgot password flow:
    - Expected: reset request succeeds (or success feedback shown) without forbidden-key error.

## 5) Risks/Notes
- Existing local `.env` may still contain a leaked secret key; code changes alone do not fix this until anon/publishable key is used.
- Secret key rotation/revocation is a required manual action in Supabase dashboard if exposure occurred.
- `prestart` validation may initially block stale local env setups; this is intentional fail-fast behavior.
- After env updates, Expo cache restart (`--clear`) is required to avoid stale-process false negatives.
